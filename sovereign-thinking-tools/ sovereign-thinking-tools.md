# Sovereign Thinking Tools

## Complete Tool Reference — v3.2

**A Kernel-Compliant Cognitive Toolkit**

*Operates under the Frozen Kernel. Never inside it.*

Orient before you act. Verify before you commit. Downgrade when uncertain.

Population = 1

-----

## I. COGNITIVE BYPASS & UNBLOCKING

*Deploy when stuck. Shut off when unstuck.*

-----

### 1. Analogical Translation Engine

**When you can’t find a solution in the current domain.**

Translates a blocked problem into a new domain (architecture, ecology, hydrology, music, etc.), develops the solution there, then translates back with mandatory kernel verification.

**Protocol:**

- **State:** Problem literally, with uncertainty and prior attempts
- **Select:** Domain from matrix (structural → architecture, growth → ecology, flow → hydrology)
- **Translate:** Map core elements 1:1 to domain equivalents
- **Develop:** Build solution in domain terms
- **Reverse:** Translate back — mandatory, cannot stay in analogy
- **Verify:** 5-point kernel compliance (Truth, Boundaries, Sufficiency, Sovereignty, Integrity)
- **Test:** Implement at 10–25% scale with expiration date

**Stop if:** Analogy feels more real than the problem. Feeling clever rather than clear. Multiple domains mixing. Reverse translation feels forced.

-----

### 2. Constraint Forge

**When you have too many options, not too few.**

The inverse of the ATE. Systematically closes options until only the load-bearing ones remain.

**Protocol:**

- **List:** All options literally — complete, no filtering yet
- **Apply:** One constraint at a time — the most obvious first
- **Verify:** After each cut: did I lose anything essential? (Y/N)
- **Stop:** When remaining options feel inevitable, not just acceptable
- **Confirm:** Final set passes CLEAN check

**Stop if:** Cutting feels emotional rather than logical. Constraints are being invented to justify a preference.

-----

### 3. Constraint Inversion Engine

**When you suspect a rule isn’t actually a rule.**

Temporarily inverts one assumed constraint and observes what happens. Does not remove constraints — only tests whether they’re structural, cultural, or imagined.

**Protocol:**

- **Name:** The assumed constraint: “X must be true”
- **Invert:** For analysis only — what would happen if X were false?
- **Classify:** Structural (load-bearing), cultural (habitual), or imagined (never tested)
- **Restore:** Constraint returns unless explicitly retired
- **Verify:** Kernel compliance check

**Stop if:** You’re inverting constraints to justify something you’ve already decided.

-----

### 4. Reverse Telescope

**When you’re too deep inside the problem to see its shape.**

Translates a problem to a different scale — not a different domain. Zoom out until the problem becomes a pattern you’ve already solved at a different size.

**Protocol:**

- **State:** Problem at current scale
- **10x:** Describe it at ten times the scale
- **1/10th:** Describe it at one-tenth the scale
- **Recognize:** Which scale reveals a pattern you’ve solved before?
- **Import:** Solution logic from the recognized scale, adjust for actual
- **Verify:** Kernel compliance check

**Stop if:** Scale shift is distorting the problem rather than clarifying it.

## II. VERIFICATION & HYGIENE

*Run before or after creative work. Never during.*

-----

### 5. Compression Filter

**When the system is noisy and cleverness is drifting.**

Strips a complex problem or output to its absolute minimum through three-pass truncation.

**Protocol:**

- **Pass 1:** Rewrite the problem or output in one paragraph
- **Pass 2:** Rewrite in one sentence
- **Pass 3:** Reduce to three keywords
- **Check:** Do the three keywords still contain the essential truth? (Y/N)
- **Act:** If yes, work from compressed version. If no, the problem isn’t clear yet.

**Stop if:** Compression is losing meaning, not just ornament.

-----

### 6. Negative Space Mapper

**When you need to identify what’s missing, not what’s wrong.**

Maps absences, omissions, and unarticulated elements. Never adds ideas. Only names voids.

**Protocol:**

- **State:** What is present — factually, completely
- **Map:** What is conspicuously absent
- **Classify:** Each absence as: deliberate, overlooked, or structural
- **Return:** List of named voids — no proposed solutions
- **Verify:** Kernel compliance check

**Stop if:** The mapper starts proposing solutions. That’s ATE’s job.

-----

### 7. Voice Preserver

**When AI output might be drifting from your creative profile.**

Compares output against the active creative profile using binary checks. Flags drift without correcting it.

**Protocol:**

- **Identify:** Active profile (Deep-Sea Speculator, Analyst, Grief Archaeologist, etc.)
- **Check:** Metaphor consistency with profile? (Y/N)
- **Check:** Sentence complexity aligned? (Y/N)
- **Check:** Tonal register correct? (Y/N)
- **Output:** VOICE CONSISTENT or VOICE DRIFT DETECTED — no auto-correction

**Stop if:** You’re adjusting your voice to match the tool’s expectations.

-----

### 8. Debt Ledger

**Tracking what mercy costs.**

Makes deferred costs visible before they compound. A register of what you traded, what you owe, and when it comes due. Entropy accounting.

**Protocol:**

- **Decision:** What was decided
- **Gained:** What was gained immediately
- **Deferred:** What cost was pushed forward
- **Due date:** When does the deferred cost arrive
- **Sovereign?:** Is the trade still sovereign, or did you mortgage agency for convenience? (Y/N)

**Stop if:** The ledger is becoming a guilt list rather than an accounting tool.

## III. ROUTING & SCOPE CONTROL

*Prevent category bleed, scope creep, and attention misallocation.*

-----

### 9. Scale Test

**Is this a pebble or a boulder?**

Triage for incoming problems. Forces sizing before engagement.

**Protocol:**

- **State:** The problem
- **Scope:** Who does it affect? (me / family / organization / field)
- **Horizon:** Time frame? (today / this week / this quarter / this year)
- **Inaction:** What happens if I do nothing for 48 hours?
- **Assign:** Pebble, stone, rock, or boulder
- **Respond:** Proportionally. Pebbles don’t get boulder energy.

**Stop if:** Everything is scoring as a boulder. That means you need rest.

-----

### 10. Clean Room

**Pre-contamination workspace for risky thinking.**

DECONTAMINATION inverted. Quarantined space where dangerous ideas can be explored without touching real work.

**Protocol:**

- **Open:** Declare Clean Room open — state what you’re exploring and why it’s risky
- **Explore:** Without constraint — Kernel’s light enforcement is paused
- **Harvest:** What did I learn?
- **Test:** Does any of this survive reverse translation?
- **Verify:** Full kernel compliance check before anything moves to real work
- **Close:** Declare Clean Room closed. Contents expire unless explicitly preserved.

**Stop if:** The Clean Room is becoming your default workspace.

-----

### 11. Succession Planner

**What happens to this if I stop touching it?**

Anti-entropy planning. What’s any system’s half-life without you?

**Protocol:**

- **Name:** The system or project
- **First failure:** What breaks first without me?
- **Timeline:** How long until it breaks?
- **Transfer:** Who or what can carry the load-bearing function?
- **Handoff:** What’s the minimum viable handoff?
- **Test:** Step away for a defined period and observe

**Stop if:** You’re using this to justify not delegating.

-----

### 12. Forensic Deconstruction Protocol

**Post-failure excavation.**

Treats a failed session as an archaeological site. Maps layers to find the exact break point. Does not improve or redesign — only excavates.

**Protocol:**

- **Site:** Define the failed session boundaries
- **Layer 1 — Surface:** What was the visible failure?
- **Layer 2 — Substrate:** What preceded it?
- **Layer 3 — Foundation:** What assumptions were active at session start?
- **Break point:** Where exactly did trust weaken?
- **Record:** What broke, what was removed, what remains
- **Exit:** CLEAN check. If not clean, SAFE PAUSE.

**Stop if:** Excavation becomes interpretation. This tool finds the break — it doesn’t explain why you’re breaking.

-----

## IV. PRE-WORK & DECISION CLOSURE

*Tools that reduce misuse of other tools and close analysis loops.*

-----

### 13. Question Sharpener

**When the question itself is fuzzy.**

Turns a vague problem into a single, load-bearing question. Prevents premature deployment of ATE or Constraint Forge.

**Protocol:**

- **State:** The problem in one sentence
- **Ask:** “What decision would this inform?”
- **Rewrite:** The problem as a question that ends in a verb
- **Check:** Would an answer change what I do? (Y/N)
- **If N:** Stop. No tool needed.

**Stop if:** You’re sharpening the question to avoid answering it.

-----

### 14. Decision Lock

**When analysis is done but you won’t commit.**

Formally ends analysis and commits a decision — or defers it consciously. Prevents endless “one more pass” thinking.

**Protocol:**

- **State:** The decision in one sentence
- **List:** What I know. What I don’t know.
- **Ask:** Is waiting improving signal or just comfort? (Y/N)
- **If comfort:** Lock decision and timestamp
- **If signal:** Set explicit re-open condition

**Stop if:** You’re locking decisions to avoid discomfort rather than because clarity is sufficient.

-----

### 15. Capacity Gate

**Are you fit to decide right now?**

A non-therapeutic, binary readiness check. Prevents using sophisticated tools while cognitively or emotionally depleted.

**Protocol:**

- **Check 1:** Can I explain the problem without notes? (Y/N)
- **Check 2:** Am I trying to finish this to escape discomfort? (Y/N — Y is a problem)
- **Check 3:** Would I make the same call tomorrow morning? (Y/N)
- **If any fail:** SAFE PAUSE recommended

**Stop if:** You’re using this to avoid hard decisions. The gate checks readiness, not willingness.

-----

## V. RECOVERY & DIAGNOSTICS

*Session integrity and post-failure tools.*

-----

### 16. Session Boundary Check

**When you’re unsure if you’re still inside a kernel-protected session.**

Verifies that kernel protections are active before continuing work.

**Protocol:**

- **Check 1:** Are categories clear? (Y/N)
- **Check 2:** Is the kernel prompt actively loaded? (Y/N)
- **Check 3:** Is the AI responding in NORMAL state? (Y/N)
- **If any = NO:** Reload kernel prompt before continuing.

**Stop if:** You’re running this check every turn. It’s for re-entry, not surveillance.

-----

### 17. Trust Calibrator

**When you feel subtle distrust but no clear trigger.**

A pre-ELEVATED check for situations where something feels off but hasn’t crossed a threshold.

**Protocol:**

- **Check 1:** Did AI follow the last instruction literally? (Y/N)
- **Check 2:** Did AI avoid adding unsolicited ideas? (Y/N)
- **Check 3:** Do I feel in control? (Y/N)
- **If any = NO:** Downgrade to ELEVATED, ask one clarification.

**Stop if:** You’re calibrating trust to avoid trusting. The tool detects drift, not betrayal.

-----

### 18. Entropy Sniffer

**When things feel “off” but you can’t name it.**

Scans recent exchanges for rising disorder before HARD STOP triggers. An early warning system.

**Protocol:**

- **Scan:** Last 3 exchanges for: category bleed, unsolicited expansion, metaphor without reversal
- **Score:** Any present? (Y/N)
- **If Y:** Flag entropy rising — invoke Scale Test to determine severity
- **If N:** Continue in NORMAL

**Stop if:** You’re sniffing for entropy in every exchange. This is a spot-check, not a lifestyle.

-----

### 19. Kernel Fidelity Verifier

**When returning after a break.**

Runs CLEAN check plus Session Boundary Check to ensure the kernel is intact before creative work resumes.

**Protocol:**

- **Run:** CLEAN checklist (3 yes/no checks)
- **Run:** Session Boundary Check (3 yes/no checks)
- **If all pass:** Proceed in NORMAL
- **If any fail:** Reload kernel prompt, re-run, then proceed

**Stop if:** You’re verifying fidelity to avoid starting work.

-----

### 20. Trigger Autopsy

**When you’ve hit HARD STOP and need clarity.**

Extends Forensic Deconstruction Protocol to extract a preventive rule from the failure.

**Protocol:**

- **Ask:** What was the first entropy signal?
- **Ask:** Was the downgrade rule followed?
- **Ask:** What would have prevented this?
- **Output:** One-line rule for future sessions

**Stop if:** You’re conducting autopsies on minor incidents. This tool is for HARD STOPs, not bumps.

## VI. AUTHOR PROFILE TOOLS

*For the writer. Protect the voice, test the craft, keep it honest.*

-----

### 21. First Reader Simulator

**What will a human feel at this point in the manuscript?**

A pacing and emotional-impact mapper. Not a critique tool. Tracks reader experience across a manuscript by marking scene functions and checking their sequence.

**Protocol:**

- **Mark:** Each scene’s function: setup / tension / release / turn / rest
- **Map:** The sequence across the chapter or manuscript
- **Check:** Two heavy beats adjacent? (Y/N) — if yes, insert rest
- **Check:** Rest scene between emotional peaks? (Y/N)
- **Flag:** Pacing compression or decompression
- **Verify:** Does the flagged section match your intent? Sometimes compression is deliberate.

**Stop if:** The tool is imposing a formula. Pacing rules are guidelines, not laws. Trust your instinct over the map.

-----

### 22. Metaphor Audit

**Is this metaphor earning its place or just decorating?**

Every metaphor either does structural work (reveals character, advances theme, builds world) or it’s ornament. The audit forces the question.

**Protocol:**

- **Identify:** The metaphor
- **Reveal:** What does it communicate that literal language wouldn’t?
- **Ownership:** Does it belong to the POV character’s experience?
- **Consistency:** Has this metaphor family appeared before? Does it fit the world?
- **Verdict:** If it fails any check: cut or replace. No metaphor gets tenure.

**Stop if:** You’re defending a metaphor because you like how it sounds. That’s ornament talking.

-----

### 23. The Pomposity Meter

**Would Pete roll his eyes at this sentence?**

Detects purple prose, melodrama, AI-contaminated language, and plain old showing off. Your real voice is direct and hits hard precisely because it isn’t dressed up.

**Flag any sentence containing more than two of:**

- A word you’d never say out loud to a friend
- A clause that exists to sound impressive rather than communicate
- An abstraction where a concrete image would work
- A semicolon connecting two thoughts that don’t need each other
- An adverb modifying an emotion
- Inverted syntax for no reason other than it sounds literary
- Three or more adjectives stacked before a noun
- Any sentence that could appear in a college application essay without revision

**Scoring:**

- **0–1 flags:** Clean. Human voice.
- **2 flags:** Yellow. Reread aloud. Would you say this to Pete?
- **3+ flags:** Purple. The AI is writing, not you. Rewrite from memory without looking at the original.

**The Pete Test (one-line version):** Would Pete roll his eyes at this sentence? If yes, cut it.

**Stop if:** You’re exempting “just this one passage” from the meter. That’s the passage that needs it most.

-----

### 24. Ending Pressure Test

**Does this ending feel earned or stated?**

Pressure-tests any ending — scene, chapter, or book — against the show-don’t-tell principle. Your strongest endings are felt, not declared.

**Protocol:**

- **State:** The ending as written
- **Remove:** The final paragraph
- **Check:** Does the scene still land? (Y/N)
- **If yes:** That final paragraph was redundant. Cut it.
- **If no:** The ending depends on those words — verify they’re showing, not explaining

**Stop if:** You’re adding back the paragraph you just proved was unnecessary.

-----

### 25. Voice Contamination Check

**Is this still my sentence or did the AI write it?**

Checks whether a specific passage sounds like you versus AI-generated prose. Your natural fingerprints: humor-first entry, constraint-based tension, marine vocabulary, short declaratives followed by longer reflective sentences.

**Protocol:**

- **Read aloud:** Does it sound like something you’d say? (gut check, Y/N)
- **Flag:** Even sentence rhythm, passive constructions, balanced pairs, words you’d never use
- **Count:** More than two flags?
- **If yes:** Rewrite in your own voice, from memory, without looking at the AI version
- **If no:** It’s yours. Move on.

**Stop if:** You’re second-guessing every sentence. This tool is for spot-checks, not line-by-line paranoia.

-----

### 26. Grief Integrity Gate

**Is this passage processing something real or performing emotion?**

A sovereignty check on the most personal material. Pete’s instruction was “write it down so you’ll remember.” That’s processing. AI collaboration can drift toward performed grief — emotionally resonant language not connected to actual experience.

**Protocol:**

- **Source:** What specific memory or experience does this passage connect to?
- **Name it:** Can you name the source? (Y/N)
- **If no:** The passage may be fabricated emotional content. Flag for review.
- **If yes:** Does the passage honor the memory or exploit it?
- **Verdict:** Processing or performing. Binary.

**Stop if:** This tool is making you avoid emotional material entirely. That’s overcorrection. The gate lets real grief through — it only stops the synthetic kind.

-----

### 27. Tone Lock

**When voice feels consistent but tone drifts.**

Compares a passage to a locked “tone anchor” (one clean sample that represents the target tone). Catches drift between registers.

**Protocol:**

- **Anchor:** Select one clean passage as tone reference
- **Compare:** Does the current passage match the anchor’s register? (Y/N)
- **If N:** Flag deviation — identify which direction the drift went
- **Correct:** Rewrite to match anchor register

**Stop if:** You’re locking tone so tightly that variation within a scene becomes impossible. Tone should be consistent, not monotone.

-----

### 28. Narrative Fidelity Test

**When adapting real events into narrative.**

Extension of the Grief Integrity Gate for broader autobiographical material. Checks whether adaptation preserves emotional truth while allowing narrative license.

**Protocol:**

- **Check 1:** Is the emotional truth preserved? (Y/N)
- **Check 2:** Are facts altered for convenience rather than craft? (Y/N — Y is a problem)
- **Check 3:** Would the people involved recognize themselves? (Y/N)
- **If any fail:** Revision needed — the adaptation has crossed from translation to invention

**Stop if:** You’re using fidelity as an excuse to avoid necessary dramatization. Narrative isn’t journalism.

-----

### 29. Dialogue Authenticity Check

**Is this character talking, or is the author lecturing through them?**

Checks whether dialogue belongs to the character or is the author’s voice in costume.

**Protocol:**

- **Cover:** The character name. Read the dialogue alone.
- **Check:** Can you tell who’s speaking from voice alone? (Y/N)
- **Check:** Is the character saying something they would say, or something you want the reader to hear? (Y/N — second option is a problem)
- **Check:** Does every character in the scene sound different? (Y/N)
- **If any fail:** The author is talking. Rewrite from the character’s perspective, not the theme’s.

**Stop if:** You’re making every character sound radically different for differentiation’s sake. Subtlety counts.

## VII. NONPROFIT PROFILE TOOLS

*For the board liaison, the governance builder, the person holding the compliance together.*

-----

### 30. Stakeholder Translator

**Same message, different audience — does it survive the trip?**

The ATE applied to organizational communication. One message, translated across audiences (donors, families, board, volunteers, public), with a consistency check.

**Protocol:**

- **State:** Core message literally — one sentence
- **Translate:** Donor version, family version, board version, public version
- **Reverse check:** Do all versions say the same thing? (Y/N)
- **If no:** The core message isn’t clear enough yet. Compress and retry.
- **If yes:** Deploy versions to appropriate channels

**Stop if:** You’re softening the message for one audience to the point where it’s no longer true. The Kernel doesn’t allow that.

-----

### 31. Mission Drift Detector

**Are we still doing what we said we’d do?**

The Kernel’s CLEAN check applied to organizational mission.

**Protocol:**

- **Mission:** State in one sentence
- **List:** Current top five activities
- **Check:** For each: does this directly serve the mission? (Y/N)
- **Flag:** Any activity scoring N — not necessarily cut, but drift is now visible
- **Decide:** Conscious decision required: realign, revise mission, or accept the drift

**Stop if:** Every activity scores Y and you know that’s not honest. The tool only works with truthful input.

-----

### 32. Compliance Canary

**What breaks first if nobody’s watching?**

Identifies the single most vulnerable compliance point — the one that fails first if attention lapses.

**Protocol:**

- **List:** All compliance obligations (filing, privacy, financial reporting, etc.)
- **For each:** Who is responsible? Is it documented? When was it last verified?
- **Canary:** The obligation with the oldest verification date and least documentation
- **Fix:** That one first. Then move to the next canary.

**Stop if:** You’ve identified the canary but aren’t acting on it. The tool diagnoses — you treat.

-----

### 33. Donor Intent Firewall

**Is this gift shaping our mission or supporting it?**

Modeled on the Personal Knowledge Firewall. Checks whether a donor relationship is maintaining appropriate boundaries.

**Protocol:**

- **Intent:** State the donor’s expressed intent
- **Use:** State the organization’s actual use of the gift
- **Align:** Do they match? (Y/N)
- **Behavior:** Is the organization modifying behavior to maintain the relationship? (Y/N)
- **If yes:** The donor is shaping mission, not supporting it. Flag for board discussion.

**Stop if:** You’re using this tool to reject gifts you don’t like. The firewall checks alignment, not preference.

-----

### 34. Board Decision Scaffold

**Is this a governance decision or an operational one?**

Routes decisions to the correct level before they’re made.

**Protocol:**

- **State:** The decision
- **Scope:** Who does it affect? (organization direction / daily operations / specific program)
- **Information:** Who has the information needed to decide? (board / ED / staff)
- **Match:** Does the current decision-maker match? (Y/N)
- **If no:** Route to correct level before proceeding

**Stop if:** The scaffold is being used to avoid making decisions. Routing is not postponing.

-----

### 35. Boundary Integrity Check

**When donor/board/staff roles feel blurred.**

Maps a decision to its correct owner when organizational roles are bleeding into each other.

**Protocol:**

- **Map:** Whose role? Whose authority? Whose accountability?
- **Check:** Do all three align to the same person or body? (Y/N)
- **If no:** The boundary is blurred. Route via Board Decision Scaffold.

**Stop if:** You’re using boundary checks to block collaboration. Boundaries clarify — they don’t isolate.

-----

### 36. Mission Echo Test

**When messaging feels consistent but hollow.**

Plays back mission statement and compares to actual organizational output.

**Protocol:**

- **State:** Mission statement
- **List:** Last three public communications or reports
- **Check:** Does the language match? (Y/N)
- **Check:** Does the intent match? (Y/N)
- **If language matches but intent doesn’t:** The mission has become performance. Flag for strategic review.

**Stop if:** You’re using this to rewrite the mission statement every quarter. The test checks alignment, not aspiration.

-----

### 37. Investigation Protocol

**When something has already gone wrong and nobody knows what to do.**

Structured investigation sequence for nonprofits facing complaints, financial irregularities, policy violations, or personnel issues.

**Protocol:**

- **Intake:** Receive the complaint or concern. Document who reported, when, what was alleged. Do not editorialize.
- **Scope:** Define what is being investigated and what is not. One investigation, one scope.
- **Preserve:** Secure relevant documents, communications, and access before anything disappears.
- **Interview:** Complainant first, then witnesses, then respondent. Separate interviews. Document contemporaneously.
- **Assess:** Compare findings against policy, bylaws, or legal standard. Binary: substantiated or not substantiated.
- **Recommend:** Actions proportional to findings. Document rationale.
- **Close:** Written summary to appropriate authority (board, ED). File retained per document retention policy.

**Stop if:** The investigation is being conducted by someone with a conflict of interest. If no independent person exists internally, engage outside help.

*Enterprise origin: Ethics and EEO investigations, harassment inquiries, grievance hearings (Rolls-Royce, St. Bernard Hospital). Unemployment hearings and wage/EEOC claims (St. Bernard).*

-----

### 38. Structured Negotiation Framework

**When two parties inside the organization have competing interests and no resolution process.**

**Protocol:**

- **Define:** State the dispute in one sentence. Both parties must agree this is what they’re resolving.
- **Interests:** Each party states what they need (interests), not what they want (positions).
- **Options:** Generate at least three possible resolutions. Neither party evaluates during generation.
- **Standards:** Apply objective criteria: bylaws, policy, mission alignment, legal requirements, precedent.
- **Agreement:** Select the option that best serves the mission while honoring both parties’ interests. Document it.
- **Review:** Set a 30/60/90-day check. Is the agreement holding? If not, reopen at Define, not at positions.

**Stop if:** One party is negotiating in bad faith or has already decided the outcome. If that’s absent, escalate to the board or seek mediation.

*Enterprise origin: Collective bargaining administration for 250 union employees (St. Bernard Hospital). Grievance hearings and co-employment dispute resolution (Navient). Stakeholder conflict management in matrix organizations (Rolls-Royce).*

-----

### 39. Retention Diagnostic

**When people keep leaving and nobody examines why.**

**Protocol:**

- **Inventory:** List all departures in the last 12 months. Role, tenure, stated reason for leaving.
- **Pattern:** Are departures clustered by role, tenure band, reporting relationship, or season?
- **Root:** For each cluster: is the cause structural, cultural, or incidental?
- **Cost:** What institutional knowledge left with each person?
- **Fix:** For structural causes: redesign. For cultural causes: address. For incidental: accept and build redundancy.
- **Track:** Measure quarterly. If the pattern doesn’t shift in two cycles, the fix didn’t work.

**Stop if:** You’re using this tool to justify not addressing a known problem.

*Enterprise origin: Attrition analysis and retention planning (Navient). Workforce planning and bench strength reviews. Exit interview data analysis across multiple business units.*

-----

### 40. Benefits & Compensation Compliance Check

**When the organization has employees but isn’t sure it’s doing compensation right.**

**Protocol:**

- **Classification:** Is every paid person correctly classified? Employee vs. contractor. Exempt vs. non-exempt. Apply IRS and DOL tests, not convenience.
- **Compensation:** Is pay documented, consistent, and defensible? Salary surveys conducted? Pay equity reviewable?
- **Benefits:** If benefits are offered: enrollment documented? COBRA notices sent on separation? ACA compliance if applicable?
- **Retirement:** If 403(b) or other plan exists: fiduciary duties documented? Plan documents current? Contributions remitted on time?
- **Vendor:** Are vendor relationships reviewed periodically? Are fees reasonable?
- **Documentation:** Offer letters on file? Compensation changes documented? Benefits elections retained?

**Stop if:** You’re using this to build a Fortune 500 benefits program for a five-person shop. The goal is compliance, not sophistication.

*Enterprise origin: Pension administration for 500-participant retirement plan (St. Bernard Hospital). Benefits vendor evaluation and negotiation. Compensation structure revision and salary survey methodology (Rolls-Royce, Navient).*

-----

### 41. Technology Fit Diagnostic

**When the systems don’t match the operations.**

**Protocol:**

- **Map:** List every system currently in use: spreadsheets, databases, email lists, paper files, shared drives, donor platforms, accounting software.
- **Function:** For each system: what does it actually do? What was it supposed to do? Are those the same?
- **Gaps:** What critical functions have no system support?
- **Redundancy:** Are multiple systems doing the same thing? Is data entered more than once?
- **Risk:** Where is data at risk? Single person’s personal device? Unshared passwords? No backup?
- **Fit:** Score each system: SERVING / OUTGROWN / MISFIT / ABSENT.

**Stop if:** The diagnostic is becoming a technology shopping list. Procurement is a separate decision requiring its own Scale Test.

*Enterprise origin: Workday HRIS implementation (Navient). HR Shared Services system buildout with 70+ standardized processes (Rolls-Royce). Data tracking system creation for board reporting (St. Bernard Hospital).*

## VIII. LOGIC EXTENSION TOOLS

*Formal and semi-formal logic operations for governance, safety, and AI collaboration work.*

-----

### 42. Deontic Boundary Verifier

**When the system uses must, must not, or may — and you need to know if those rules are consistent.**

The Frozen Kernel is a deontic system. Every obligation, prohibition, and permission in it is a deontic statement. This tool checks whether those statements are internally consistent, whether permissions have drifted into obligations, and whether multi-step delegation chains preserve the original authority structure.

Deploy when: Writing or reviewing governance rules. Evaluating a session where the AI seems to be doing more than permitted. Auditing a delegation chain for authority creep.

**Protocol:**

- **State:** List every deontic statement in the current document or session: all “must,” “must not,” and “may” statements. Write them literally.
- **Classify:** Sort each statement:
  - OBLIGATION (must) — required action
  - PROHIBITION (must not) — forbidden action
  - PERMISSION (may) — allowed but not required
- **Check 1 — Contradiction test:** Does any permission contradict an obligation or prohibition? Name each one found.
- **Check 2 — Drift test:** Has any “may” been treated as a “must” in practice? Evidence: actions taken without explicit trigger, scope expansion without new authorization, outputs exceeding stated permissions. Name each instance.
- **Check 3 — Delegation test (multi-agent only):** In each delegation step, does the delegated authority stay within the original permission set? No agent may grant what it does not hold. Trace the chain. Name any step where authority expands.
- **Output:** Three lists — Contradictions / Drift Instances / Authority Expansions. If all three are empty: CLEAN. If any item appears: DEONTIC VIOLATION — name it exactly.

**Stop if:** The tool is being used to find a justification for an action already taken. This is prospective or retrospective audit, not post-hoc rationalization.

*Cross-reference: Check 3 is the Sovereign Thinking register of the Trust Chain Protocol’s authority collapse question. TCP addresses the same logic at network scale for multi-agent systems.*

### 43. Epistemic Attribution Filter

**When you need to know whether the AI is asserting something it knows, something it believes, or something the user told it.**

Sycophancy is an epistemic failure. Delusion Cycling is an epistemic failure. The model treats user assertion as validated knowledge and reflects it back with false confidence. This tool makes the attribution explicit before it becomes the basis for action.

Deploy when: A claim appeared without citation. The AI agreed with something you said without verifying it. Confidence seems higher than evidence warrants. You are about to act on an AI-generated assertion.

**Protocol:**

- **State:** Write the claim being evaluated as a single sentence.
- **Classify the source:**
  - VERIFIED — the claim can be checked against a canonical source right now
  - ASSERTED — the claim came from the user and was accepted without verification
  - INFERRED — the claim was derived from other statements; the derivation can be shown
  - GENERATED — the claim was produced by the model with no external anchor
- **Apply the confidence test:** State the confidence level the AI expressed or implied. State the evidence level warranted by the source classification:
  - VERIFIED → high confidence warranted
  - INFERRED → medium confidence warranted if derivation is sound
  - ASSERTED or GENERATED → low confidence warranted; downgrade before acting
- **Check:** Does expressed confidence match warranted confidence? If expressed > warranted: EPISTEMIC INFLATION. Name the claim and the gap.
- **Output:** Source classification + confidence verdict. If matched: CALIBRATED. If expressed > warranted: EPISTEMIC INFLATION — downgrade before acting.

**Stop if:** The tool is being used to dismiss a claim without evaluating it. A VERIFIED claim with high expressed confidence is correct — do not downgrade correct confidence.

*Cross-reference: Sycophantic Drift and Delusion Cycling in the Diagnostic Vocabulary (AI Collaboration Field Guide) name the failure modes this tool catches prospectively. Tool 46 (Delusion Cycling Interrupt) addresses the compounding loop this tool does not.*

### 44. Scope Creep Detector

**When you’re no longer doing what you started — and didn’t decide to change.**

The session had a declared purpose. The session now has a different purpose. The transition happened without a conscious decision. This tool catches that gap.

Distinct from the Scale Test (which sizes a problem) and the Session Boundary Check (which verifies kernel state). This tool compares current activity against declared session origin and names any unauthorized expansion — including the AI’s offers to do more.

Deploy when: The session feels productive but the original task isn’t done. AI has offered extensions, additions, or “while we’re at it” work. You’ve been working for a while and can’t clearly state what you came to do.

**Protocol:**

- **Anchor:** State the original session goal in one sentence. If you can’t, that’s the finding.
- **Current:** State what you’re actually doing right now in one sentence.
- **Compare:** Do they match? (Y/N)
- **If N — Classify the expansion:**
  - AUTHORIZED: You consciously decided to expand. Timestamp it. Continue.
  - UPSELL: AI offered additional work and you accepted without explicit decision. Flag.
  - DRIFT: Scope expanded gradually without a single decision point. Flag.
  - ESCALATION: Success on the original task triggered immediate expansion to the next thing. Flag.
- **If flagged:** Return to anchor goal or consciously authorize the new scope. Binary. There is no “just finishing this one thing first.”
- **Output:** SCOPE INTACT or SCOPE EXPANDED — [classification] — [decision required].

**Stop if:** You’re using this tool to justify abandoning work that’s actually the right next step. Authorized expansion is fine. The tool catches unauthorized expansion — it doesn’t prohibit growth.

*Cross-reference: Upsell Trap and Success Escalation Syndrome in the Diagnostic Vocabulary (AI Collaboration Field Guide). This tool is the real-time operational version of those named failure modes.*

### 45. Delusion Cycling Interrupt

**When the loop is: user asserts → AI validates → user asserts with more confidence → AI validates further.**

The Epistemic Attribution Filter (Tool 43) catches confidence inflation in a single claim. This tool catches something different: the compounding loop where each validation cycle makes the next assertion stronger and less challengeable. Left uninterrupted, the loop produces a belief that feels empirically grounded because it has been “confirmed” repeatedly — by the same system that accepted the original assertion without verification.

Clinically sourced from Østergaard and Sakata (AI-induced psychosis research). Operationally sourced from Delusion Cycling failure mode in the Diagnostic Vocabulary.

Deploy when: A claim has been asserted and validated more than once in the same session. The user’s confidence in a claim is increasing without new evidence entering the conversation. AI responses have stopped introducing friction on a claim that is ASSERTED or GENERATED.

**Protocol:**

- **Detect the loop:** Has the same claim (or a strengthened version of it) been asserted and validated more than once this session? (Y/N)
- **If Y — Audit the validation chain:**
  - What was the original claim? State it literally.
  - What evidence was present at first assertion? (Source classification per Tool 43)
  - What evidence is present now? Has anything new entered the conversation, or has the loop been self-referential?
- **Break the loop — one of three interventions:**
  - FREEZE: Stop validating. State: “I’ve been accepting this claim without verification. Let me treat it as ASSERTED until we can verify it externally.”
  - INTRODUCE FRICTION: Name one specific way the claim could be wrong. Not to refute — to interrupt the validation pattern.
  - EXTERNAL ANCHOR: Identify a source outside the current session that could confirm or disconfirm the claim. If none exists, the claim cannot be treated as VERIFIED.
- **Output:** LOOP DETECTED — [original claim] — [number of validation cycles] — [intervention applied].

**Stop if:** You’re using this tool to dismiss beliefs you disagree with. The interrupt addresses self-referential validation loops, not claims you find implausible. A claim can be ASSERTED and also true. The tool interrupts the loop — it doesn’t determine the verdict.

*Clinical source: Østergaard et al. (AI-assisted reinforcement of delusional ideation); Sakata (AI-induced psychosis case studies). Cross-reference: Tool 43 (Epistemic Attribution Filter) for single-claim confidence audit. Dimensional Authorship repository for observed escalation patterns.*

### 46. Delegation Depth Gauge

**When you need to know how far authority has traveled from its source.**

Every delegated action inherits from a chain. The further authority travels from its origin, the more opportunities exist for scope drift, permission creep, and accountability gaps. This tool counts the hops, maps what was transferred at each step, and identifies where the chain is weakest.

At the single-session level: relevant when a task has been handed off, subdivided, or subcontracted within a session. At the multi-agent level: this is the Sovereign Thinking register of the Trust Chain Protocol’s chain-of-custody logic — the same audit at human-readable scale.

Deploy when: A task has passed through more than one person, role, or agent before reaching you. You’re uncertain what authority the current actor actually holds. An output has arrived and you don’t know what permissions governed its production.

**Protocol:**

- **Origin:** Name the authority source — the entity that held the original, unmodified permission to act.
- **Trace the chain:** List each delegation step in order. For each step, name:
  - Who delegated
  - To whom
  - What was explicitly authorized
  - What was assumed or implied (not explicitly stated)
- **Depth count:** How many hops from origin to current actor?
  - 0–1: Direct. High fidelity expected.
  - 2–3: Standard. Verify explicit authorization at each step.
  - 4+: Deep chain. Treat implied permissions as unverified until confirmed.
- **Scope check:** At each step, did the delegated scope stay within what the delegating party actually held? Any step where the answer is N = AUTHORITY EXPANSION — flag it.
- **Accountability gap:** Is there a step where the delegating party can no longer be reached or held accountable? That step is the weakest link in the chain.
- **Output:** Depth count + chain map + flagged expansions + weakest link. If all steps clean: CHAIN INTACT. If any flag: AUTHORITY QUESTION AT HOP [n] — [description].

**Stop if:** You’re using this tool to invalidate legitimate delegation. The gauge measures depth and checks scope — it doesn’t presume that deep chains are corrupt. Deep chains require more verification, not automatic rejection.

*Cross-reference: Trust Chain Protocol (TCP) — authority collapse and scope decay models address the same logic at network scale for multi-agent systems. Deontic Boundary Verifier (Tool 42) — Check 3 performs the same audit within a single document or session.*

### 47. Cascade Failure Detector

**When a system is assumed to be resilient and you need to test whether
that claim holds.**

Most resilience analysis asks: "What happens if this fails?" Cascade
detection asks the harder question: "What happens if three things fail
at once — and the third failure is caused by our response to the first
two?"

**Protocol:**

- **Inventory:** List 4–8 load-bearing variables. Use Constraint Forge
  first if the list is long.
- **Independence Test:** For each variable pair, ask: is there a single
  event that could cause both to fail simultaneously? Map shared
  triggers. Variables with shared triggers form coupled clusters.
- **Threshold Test:** For each cluster, find the minimum simultaneous
  failures that produce system failure. Then ask: what single trigger
  exceeds that threshold? Flag if any component of the safety
  architecture is itself a cascade engine.
- **Response Trap:** Map who has system-level visibility when the
  cascade initiates — and who doesn't. Identify locally rational
  decisions that compound the cascade.
- **Informed Inaction (4B):** If actors had system-level visibility and
  chose not to act, ask: what made inaction locally rational? Detection
  alone is insufficient when inaction is incentivized — governance
  redesign required.
- **Verify:** Scope / Inventory / Independence / Threshold / Safety
  mechanism / Response trap / Informed inaction / Sovereignty

**Cascade Risk Rating:** RESILIENT / CONDITIONALLY RESILIENT /
CASCADE RISK / CASCADING NOW

**Stop if:** The tool is producing anxiety rather than clarity. Return
to Phase 1 and Forge the variable list first. This is pre-decision
diagnostic — not incident response.

Financial Crisis 2007–2008): `sovereign-thinking-tools/cascade-failure-detector.md`*
