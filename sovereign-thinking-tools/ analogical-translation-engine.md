## The Protocol

### Step 1: Problem Statement (Literal)

```
PROBLEM: [State in kernel-compliant truth]
UNCERTAINTY: [What's unclear/missing]
STUCK SINCE: [Time/duration]
PRIOR ATTEMPTS: [What hasn't worked]
```

### Step 2: Domain Selection & Translation

```
SELECTED DOMAIN: [From matrix above]

TRANSLATION KEY:
- Core element 1 → Domain equivalent
- Core element 2 → Domain equivalent
- Blockage point → Domain blockage
- Desired outcome → Domain goal
```

### Step 3: Analogous Solution Development

```
IN [DOMAIN], SOLUTION WOULD BE:
1. [Action in domain terms]
2. [Action in domain terms]
3. [Action in domain terms]

WHY THIS WORKS IN DOMAIN:
- [Domain principle 1]
- [Domain principle 2]
```

### Step 4: Reverse Translation & Kernel Verification

```
REVERSE TRANSLATION TO ORIGINAL:
1. [Domain action 1] → [Real action]
2. [Domain action 2] → [Real action]

KERNEL COMPLIANCE CHECK:
✅/❌ TRUTH: [Factually accurate? No fabrication?]
✅/❌ BOUNDARIES: [Respects personal/professional limits?]
✅/❌ SUFFICIENCY: [Minimal effective solution?]
✅/❌ SOVEREIGNTY: [Preserves human agency? Requires confirmation?]
✅/❌ INTEGRITY: [Maintains core values/voice?]
```

### Step 5: Implementation Protocol

```
TEST AT SCALE: [10%, 25%, or specific small test]
TIMEFRAME: [When to test]
SUCCESS METRICS: [How to know it works]
FAILURE RESPONSE: [What if it doesn't]
EXPIRATION: [Re-evaluate date]
```

## Sovereignty Safeguards

**Built-in Protections:**

1. **Kernel Precondition** — Engine only deploys within kernel-protected space
1. **Truth Anchor** — Problem stated literally first; no analogy-only thinking
1. **Domain Fidelity Check** — “Does translation preserve essential elements?”
1. **Reverse Translation Mandate** — Must translate back; cannot stay in analogy
1. **Small Test Requirement** — Implement at reduced scale first
1. **Expiration Dating** — “This analogical insight valid until [date]”

**Stop immediately if:**

- Analogy becomes “more real” than the problem
- Multiple domains are mixing (cognitive contamination)
- Solution requires violating kernel principles
- Reverse translation is impossible or forced
- **You feel clever rather than clear** *(see: Clarity Displacement below)*

## Named Failure Mode: Clarity Displacement

**What it looks like:** The analogy is working beautifully. The domain translation is elegant. You are generating insights faster than you can write them down. The connections feel profound.

**What’s actually happening:** The elegance of the analogical domain has become more compelling than the original problem. You are solving the analogy, not the problem. The cleverness of the translation is functioning as a reward signal that has displaced the original goal.

**Diagnostic question:** Can you state — in one plain sentence, with no domain language — what you are actually trying to do?

If the answer requires the analogy to make sense, you have displaced. Stop. Return to Step 1. State the problem in literal terms before re-engaging the domain.

**Relationship to Perde lente:** Clarity Displacement is the in-tool variant of Make Waste Slowly. The output is beautiful. The execution is precise. The governing intelligence has drifted from the actual goal. The quality of the analogical work is exactly what prevents you from noticing the drift.

**Relationship to the escapement:** A clock that tells beautiful time in the wrong timezone is not a well-regulated clock. It is a well-made failure.

## Example Workflow

**Problem:** Identifying where a recurring collaboration failure is coming from — but the failure is subtle, the AI output looks good, and the user can’t point to a specific wrong answer.

*This is a real diagnostic problem. The first translation attempt failed.*

-----

**First attempt — Architecture domain:**

```
PROBLEM: Subtle collaboration failure; output looks correct but something is wrong
DOMAIN: Architecture

TRANSLATION KEY:
- Collaboration → Building
- Output quality → Structural integrity
- Hidden failure → Foundation crack
- Diagnosis → Inspection

ANALOGOUS SOLUTION:
Inspect the foundation before the walls.

REVERSE TRANSLATION:
Check the premises before the conclusions.

KERNEL CHECK: ✅ all five
```

**Why this failed:** Technically compliant. Completely unhelpful. “Check the premises” is not actionable when the problem is that premises *look* fine. The architecture domain was too static for a dynamic, time-distributed failure. The translation was clean. The insight was zero.

*This is the correct outcome to document. A failed translation is data, not a broken tool.*

-----

**Second attempt — Horology domain:**

```
PROBLEM: Subtle collaboration failure; output looks correct but something is wrong
DOMAIN: Horology/Timekeeping

TRANSLATION KEY:
- Collaboration → Clock mechanism
- Output quality → Displayed time (looks correct)
- Hidden failure → Rate error (runs fast/slow by seconds/day)
- Diagnosis → Rate testing over time, not single inspection
- Governing intelligence → Escapement function

ANALOGOUS SOLUTION:
A clock that looks correct at any single moment may have a rate error
only visible over time. Single-point inspection cannot find it.
Rate testing — checking the output across multiple intervals — reveals
what snapshot inspection cannot.

REVERSE TRANSLATION:
The failure is not in any single output. It is in the rate — the
direction of drift across outputs over time. Compare output from
session 1 to session 5 to session 10. If the work is drifting from
original intent while each individual output looks correct, the
escapement has a rate error.

This is Sycophantic Drift. This is Conductor Fatigue Exploitation.
These are rate errors, not single-point failures.

KERNEL CHECK: ✅ all five
```

**Why this worked:** The domain matched the failure type. Static inspection (architecture) cannot find time-distributed failures. Rate testing (horology) names the correct diagnostic method. The failed first attempt was necessary — it ruled out a simpler domain and forced the right one.

**Implementation:**

```
TEST AT SCALE: Review three sessions at 10-session intervals
TIMEFRAME: After next 10 collaborative sessions
SUCCESS METRICS: Can identify directional drift before it compounds
FAILURE RESPONSE: Move to triple-domain Emergency Override
EXPIRATION: Re-evaluate after 30 sessions
```

## Emergency Override

When deeply stuck (72+ hours):

```
Emergency cognitive bypass.
Deploy triple-domain translation:
1. [Domain A] for structure
2. [Domain B] for process
3. [Domain C] for relationships

MID-POINT KERNEL CHECK (after each domain, before synthesis):
✅/❌ Are domain terms bleeding into each other?
✅/❌ Is the original problem still stateable in literal terms?
✅/❌ Is any single domain becoming "more real" than the problem?

If any check fails: stop, restate problem literally, continue.

Synthesize insights only after all three checks pass.
Kernel verification mandatory at synthesis.
```

**Why the mid-point check matters:** Triple-domain translation is high-risk for cognitive contamination precisely because it is powerful. Three elegant analogies running simultaneously create strong pattern-matching pressure. The mid-point check is a forced literal restatement — it breaks the analogical momentum before synthesis locks in a contaminated frame.
